A subjective WCAG 2.1 checklist on how to make a web page accessible

published: 20.12.2019, 10 minute read

My Take on WCAG 2.1

This only covers the WCAG 2.1 levels A and AA (not AAA) because in the project were I was involved we never aimed at level AAA (even W3 does not recommend that each site should meet level AAA - see also the understanding section on WCAG2.0 conformance on the w3-website). For different reasons we concluded that level AAA was not necessary for our site - and also practically not achievable. It should also be noted that this list is very subjective and it will in no way grant level AA compliancy. That was never its goal. It is more to be seen as the things that I deemed to be relevant and important when trying to make a site more accessible. I hope that it helps me as a reference guide when I need to make sites accesible in the future - and I hope that you might also profit. You're free to copy the list and use it in your own project.

Note

Please note that this is a sort of a "living document" in that I might update the list in the future if I find some points in the WCAG-documentation to be worthy of an upgrade/edit/delete. The list should stay mostly the same, but details can change.

HTML

HTML elements are used according to their specification (e.g. links should not contain any other links, forms should be used properly, etc.)
The "autocomplete"-attribute is used properly (even though Chrome seems to break it deliberatly (see this bug report in Chromium)
Buttons are used as buttons (they do not navigate) and links are used as links (they navigate)
Tabindex is used properly (basically: don't use tabindex of 1 or higher it will get you and your users into issues)
The lang-attribute is set with the correct value on the html-element
The lang-attribute is set on individual elements where necessary.

Content & Structure

Heading structure is correct - there are no "holes" in the heading levels (so not a h1, then a h3 etc.) and the sequence is logical.
Labels are used correctly with inputs (and maybe also with icons, graphics etc. when those elements need further explanation)
Each page has a proper title set
All elements, needing explanation, have a correct and descriptive text-alternative e.g. images, graphics, etc.
Elements that should not be announced by screen-readers are properly hidden (e.g. , or aria-hidden="true" on icons, etc.)
The visual order matches the DOM order. If a screen reader user moves through the page he experiences the same structure as sighted users.
The tab/keyboard order follows the sequence of the content.
There are links present that let you jump directly to the main content (skip links)
Repeated content, such as navigation, can be skipped via links
Multiple ways exist to reach a web page within a set of pages (e.g. via a navigation, sitemap, search, etc.)
Navigation that is present on multiple pages always appears in the same order.

Forms

Form fields have the correct accessible hints about their format.
Form fields are labelled correctly (labels are correctly associated with their input fields) (do not only use placeholders to describe what needs to be put in an input field)
Form validation errors can easily be navigated to, are descriptive and can be linked to the erroneous form
If input errors are detected and suggestions/corrections for the errors are known, they are provided to the user (unless it would threaten security)
If a form causes legal commitment, a financial transaction, or modifies or deletes user data on a system, the user has a chance to either reverse the submission once its made, or he can review, correct and confirm the submission (if the user data is checked for errors)

Keyboard

The keyboard focus is visible (outline: none should not happen)
Keyboard navigation is never trapped anywhere in the page
Keyboard shortcuts are prevented, or, if present, they can a) be turned off, or b) be remapped (to include also non-printable keys like ALT), or c) they are only active when the component has focus.

Design/UI

Contrast for text and interactive elements is sufficient (but is also not too high - this is actually not part of WCAG but I picked the idea up in the "Inclusive Design Patterns" book by Heydon Pickering)
Color, shape, visual location, etc. is not the sole mean to convey information, prompting an action etc. As an example don't prompt an action like "If you're interested click the red button"
Unless essential for the functionality of the page, the site is not restricted to either landscape or portrait mode
No more than three flashes flash in any one second period and the flash is below the general flash and red flash threshold (this is to prevent triggering seizures)
There is enough time to complete a process (unless time is an integral part of the process like in an auction). If there is a time limit it can be turned off or be extended
Moving, blinking, scrolling or auto-updating content that is presented in parallel to other content can be paused, stopped or hidden

Audio/Video

Audio that plays automatically for at least more than 3 seconds can be paused, stopped, or muted
Time-based media (such as prerecorded audio or video) is accessible (e.g. audio has captions)

Interactions

If the viewport is only 320px wide, there is no horizontal scroll bar
Text can be resized up to 200% without breaking the layout or the functionality of the page
Various text-properties can be changed without loosing functionality or content (e.g. line-height can be resized to 1.5 times the font-size, etc.)
Content that is shown on hover is used in an accessible way: it is dismissable, the pointer can be moved on to it without it going away, and the content is shown until the hover is removed
Focusing a component or inserting information into an input does not initiate a change of context (e.g. a pop-up window appears suddenly)
Status messages (e.g. a small success-message at the top of the screen when something was saved) are programmatically determinable so that they can be presented to the user without having to receive focus.
