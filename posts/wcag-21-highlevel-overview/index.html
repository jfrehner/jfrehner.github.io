Most important elements to consider when implementing web accessibility (based on WCAG 2.1)

published: 17.12.2019, 4 minute read

Introduction

Accessibility (or A11y - as it is often to referred to as well) is quite important for my current work. That's why I invested some time reading the actual "Web Accessibility Content Guidelines" (WCAG) 2.1 and tried to condense all the principles, guidelines and success criterions for level AA into a small list. The result is the below set of the points that I think are the ones which give the biggest return on investment. This list is certainly highly subjective and currently reflects my personal take on WCAG 2.1 and might very well change in the future.

Most important elements to consider when implementing web accessibility (based on WCAG 2.1)

  • Pay attention to your DOM structure - it should follow the visual order. As a fun experiment you can take your site, strip all css from it, and the content structure should still be the same as before and be in a meaningful way. Also make sure that the tab-order is correct. The most important take-away here is to not use a "tabindex" of 1 or greater.
  • Non-text elements have a descriptive non-text alternative. For example, images have an appropriate "alt" text, input elements are labelled correctly, etc.). As a sidenote: decorative images that should not be "hidden" from assistive technologies (such as screen readers) you can put an empty tag as the alt-text of the image - this will successfully hide the image in the accessibility-tree.
  • Use HTML-elements correctly, perhaps most importantly: use buttons as buttons and links as links and don't mix them up or use divs to create buttons... HTML is by default accessible and provides a lot out of the box that has to be recreated when we use/repurpose div's to buttons, checkboxes, etc..
  • Pay attention to contrast: contrast between interactive elements, links and text and its background should be sufficient.
  • Don't disable zoom (via the meta-tag) and make sure that your web page stays usable when content is zoomed (for example via browser-zoom)
  • Do not hide the keyboard focus. As a keyboard-user, it is quite confusing when you press tab multiple times but are not able to locate the focus on the web page that you read...
  • Make forms accessible. This could be its own article as this can get quite complex and difficult to achieve, but the most important points here are:
    • Correctly label form-fields
    • Provide accessible hints for an input-field and preferrably provide them before the input-field so that a user of assistive technology knows what format he needs to insert into the input-elements once he gets there.
    • Form validation errors should be easily to navigate to, are descriptive (don't just say "Colorname is invalid" be more descriptive e.g. "Colornames cannot contain any numbers"), and can be linked to the form that is erroneous.
    • If a form submission would trigger actions that might have legal, financial (e.g. you buy something), or data (e.g. deleting an account) consequences, make sure at least one of the followin is true:
      • the submission is reversible
      • the data is checked for input error and a possibility exists to correct the errors before submitting the form
      • the inserted information can be reviewed, confirmed and corrected before triggering a submission
  • WCAG 2.1 defines four areas which can prevent a whole web page of being accessible. So pay attention that all of the following points are taken care of:
    • There should be no keyboard trap. A user should always have the possibility to tap out of an element when he can tap into it.
    • There should be no flashes that are known to cause seizures (e.g. never let anything flash three times in any one second period - if you're not a crazy advertiser this should not be a problem, I think?)
    • If content automatically moves, blinks, scrolls or auto-updates and is presented in parallel with other content, it can be stopped, paused or hidden.
    • If you have audio on your page that automatically plays longer than 3 second, you should provide a mechanism to either pause or stop the audio, or to adjust the audio volume independently from the overall system volume.
